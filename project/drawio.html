Explore Music by Drawing Album Art

Context: In this class project for software engineering class, we are building a data visualization framework using Java. Inspired by panther.io, Vanessa, Sean, and I wanted to create a framework that would be useful for network visualization. However, we don’t want the visualization to be limited to music exploration; instead, we want it to be a generic framework that can visualize all kinds of network in general. Some examples would be artwork, book, or articles that are connected by edges that identify how these creations are similar or linked to each other.

Prior to development: After the idea was finalized, we sat down together to finalize the architecture of the framework. The framework we designed has two plugins that users needed to provide: data plugin and UI plugin. In the data plugin, the client would have to specify linkage of current node to next node and how to get relevant description of current node. While in the UI plugin, the user have the liberty to configure components that are rendered in the display panel.

During our discussion, we have made separation of front and back-end our priority. To ensure that our roles are not intertwined in the project, we needed to ensure the front and back end are not highly coupled. We therefore drafted a detailed UML diagram of classes in the framework, as well as the callback functions that the plugin needs to provide.

We therefore split into three roles: Sean was responsible for writing the back-end of the framework, and I merrily took up the front-end part of the framework (specifically UI and data visualization), while Vanessa took care of writing sample plugins for testing and demo purposes.

Process: During the research phrase, I was looking into data visualization libraries in Java that would be useful in visualizing network. However, libraries like JChart are not very customizable and aesthetically pleasing. My quest turned out to be rather fruitless. I then began to look for ways to integrate a web front-end to a Java backend, since HTML/CSS/JS are the best in constructing a highly interactive, customizable UI, not to mention the powerful D3.js library that can help with visualization.

This is when JavaFx came into play. JavaFx is a browser that can receive and emit commands from and to Java backend, while at the same time render web pages like normal browsers do. I was thrilled — but Sean and Vanessa were not so much. They of course love the front-end to be pretty, but JavaFx was definitely not our safest bet, given that communication between Javascript and Java introduced an extra layer of uncertainty. We were not sure if the front-end would be completed within given time.

To make sure the idea was worth pursuing, we did a mini test page on the JavaFx, and it was able to listen to command in Java to render a dummy page. I then dived into developing the front-end for the framework, while Sean and Vanessa into their own domain.

This was my very first few encounters with D3.js. I took a lot of network graph examples from bl.ock.org, and toggled a few lines of code here and there. The result of doing interactive visualization with D3.js was a very glitchy UI: some nodes on the graph did not move even though the UI event had called for update. There was an extra layer of complication since we wanted to specifically track that which node has been clicked and update the position of the graph, as well as appending new nodes. In retrospect, D3 was not the optimal choice for the visualization; in fact, a normal SVG graphic could have suffice in the case in our framework. I was glad that despite all the issues, the UI turned out quite well and was not a major roadblock in our project.

There were quite a few unforeseen problems during development, but the real nightmare began when we started to integrate the Java backend and Javascript frontend. When some parts of the UI are not rendering, Sean and I had to put debug statement in all bridges between the Javascript and Java to find out from where the communication was failing. Some of these statements were printed in the Eclipse (Java IDE) console, while some are in the browser. There were also indeterministic behaviors on the page, since some data structure were initialize after they were called for. The sequence between web and backend required particular careful planning.

During the whole time when we were developing, we carefully monitored our working progress on the white board, carefully planned out the public interfaces for both the backend and the frontend. We were glad that we were able to finish the project before the chime of midnight, and we were really proud of how it turned out. We’re hoping to upload a demo soon!
